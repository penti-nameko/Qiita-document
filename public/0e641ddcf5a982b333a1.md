# 2. そもそもk8sとは
簡単に言えばシステムが死んだら作り直して正常な状態をキープするシステムです。
もう少し噛み砕くと、「コンテナをいくつもまとめて、落ちたら勝手に作り直して、必要なら勝手に増やしてくれる仕組み（コンテナオーケストレーションツール）」というのがk8sの正体です。

このk8sを動かすには当然「土台」となるマシンが必要になるので、まずはそこを整理していきます。

# 3. マシンの種類と活用
マシン、と一言でいっても当然種類があります。
物理マシン、仮想マシン.....その中にも種類が....というとわかりにくいので
ざっくり分けると

物理マシン：我々の言う物理サーバーやPC
仮想マシン：ホストとなる物理マシンの上で運用される仮想的なコンピューター

となります。
要は何が言いたいかというと、

## 実態があるかないかです
k8sクラスタ（複数のマシンをまとめて1つのk8s環境として動かす集まりのこと）を作る上ではあまりどちらでも問題はないですが、仮想マシンのほうが拡張性はあります。
当然、ニーズに応じて増やしたり減らしたりしやすいからです。

とはいえ、k8sはエラーで止まったコンテナ等を容赦なく作り直す仕組みなので、何も対策しないとコンテナの中に置いていたデータも一緒に消えてしまいます。
なので、データを残したい場合はPersistentVolumeなどを使ってデータの保存先をコンテナの外に切り出しておく必要があります。
これは「物理マシンか仮想マシンか」とは別軸の話で、どちらを選んでもこのデータ保持の設定自体は必要になります。物理マシンにしたから安全、仮想マシンだから危ない、という単純な話ではない点は覚えておくといいと思います。

# 4. 仮想化の種類
じゃあ実際に仮想化するとなるとどういうシステム形態になるのかというと
![image.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/4434251/e27a0a53-9aa1-4d1d-86fb-1d8c5c6b4e57.png)
方式がこの三種類になります。

左がホスト型といい、
Oracle VM VirtualBox等のWindows,Mac,Linux等で幅広く使いやすく、扱いやすい形になります。

中央はハイパーバイザー型というもので、ホストOSを薄く（あるいはほぼ介さずに）動かす分、ホスト型よりリソースを効率的に使えます。
主な例はVMware ESXiやMicrosoft Hyper-Vなんかがそれにあたります。

そして残ったコンテナ型は、ホストのOS(カーネル)を共有しつつ、アプリケーションごとに必要な部分だけを切り出して動かす方式です。ゲストOSを丸ごと持たない分起動が速く、リリース期間の短縮なども可能になります。
主な例として、DockerやKubernetesがまさにそれです。
